# 한국어 버전

## 문서 목적
이 문서는 `SMART 조달청 계산기`의 Phase 4 운영 제품화 범위를 정의합니다.

Phase 4는 판단 엔진 자체를 더 똑똑하게 만드는 단계가 아니라, 실제 관리자가 로컬 PC에서 매일 운영할 수 있도록 상태 확인, 작업 관리, 실패 복구, 백업/복원을 제품 기능으로 만드는 단계입니다.

이번 세부계획서는 우선 다음 범위만 다룹니다.

- Phase 4A: 운영 대시보드
- Phase 4B + 4C: 작업/실패 관리
- Phase 4D: 백업/복원

## 현재 기준선
2026-05-24 기준으로 현재 시스템은 다음 기능을 이미 갖고 있습니다.

- 일반 프로젝트 문서 업로드/분석
- 법인 증빙자료 업로드/추출/분류/승인 기반 프로필 보강
- 나라장터 API 기반 공고 검색/저장/첨부 다운로드/요약
- 공고 요구조건 후보 추출과 법인 부족조건 미리보기
- 기준문서 업로드/파싱/OCR degrade/청킹/로컬 인덱싱/검색
- 기준문서 규칙 후보 추출/승인/반려/수정
- Phase 3 부족조건 중심 판단 실행, citation 후보, 준비 가이드, 검토 상태
- 나라장터 자동 수집 실행/이력 화면
- PDF 샘플 캐시 기반 QA 구조

따라서 Phase 4는 새로운 판단 로직보다 운영자가 다음 질문에 바로 답할 수 있게 하는 것이 핵심입니다.

- 지금 시스템이 정상인가?
- 최근 실패한 작업은 무엇인가?
- 실패 사유와 재시도 방법은 무엇인가?
- 중요한 데이터가 백업되어 있는가?
- 복원 가능한 백업인지 확인할 수 있는가?

## 구현 진행 현황

2026-06-07 기준 Phase 4A 운영 대시보드, Phase 4B+4C 작업/실패 관리, Phase 4D 백업/검증/복원계획 dry-run의 1차 구현을 완료했습니다.

완료된 항목:

- `GET /api/operations/summary` 운영 요약 API
- 운영 상태 집계 서비스
- 실패 작업 집계
- 검토 대기 큐 집계
- DB/storage/OCR/나라장터 API/AI provider 상태 표시
- API 키 원문 비노출 테스트
- `/operations` 운영 대시보드 UX
- 브라우저 렌더 확인
- Phase 4B `operation_runs` 공통 작업 이력 테이블
- `GET /api/operation-runs`, `GET /api/operation-runs/:id`, `POST /api/operation-runs/:id/retry`
- 나라장터 수집, 판단 실행, 기준문서 재처리, 기준 규칙 후보 추출의 작업 이력 기록
- `/operation-runs` 작업/실패 관리 UX
- Phase 4D `backup_runs` 백업 이력 테이블
- `GET /api/backups`, `POST /api/backups`, `POST /api/backups/validate`, `POST /api/backups/restore-plan`
- 백업 ZIP 생성, manifest/checksum 검증, `.env` 제외 검증
- `/backups` 백업/복원 dry-run UX
- 실행 중 SQLite DB는 직접 압축하지 않고 sqlite backup snapshot을 사용합니다.
- 백업 실패 payload는 validation errors 기본 구조를 유지해 프론트 런타임 오류를 방지합니다.
- 운영 대시보드의 백업/실패 요약은 `backup_runs` 상태를 반영합니다.

남은 항목:

- 작업 이력 대상을 일반 문서 분석/증빙자료 분석까지 확장
- 실제 파일 교체 복원 기능은 사용자 승인 후 Phase 4D 후반 또는 Phase 4.5로 분리
- 백업 저장 위치 선택 UX
- 백업 자동 주기 실행은 별도 승인 후 확장
- 인증/권한/다중 사용자/클라우드 배포는 Phase 4.5 또는 Phase 5로 분리

## 제품화 원칙

1. 로컬 단일 PC 운영 가정을 유지한다.
2. 인증, 다중 사용자, 클라우드 운영은 Phase 4 기본 범위에 넣지 않는다.
3. 실패를 숨기지 않고 관리자에게 명확히 노출한다.
4. 자동 재시도보다 수동 확인 가능한 재시도를 먼저 구현한다.
5. DB와 파일 저장소를 함께 백업한다.
6. 백업/복원은 파괴적 작업이므로 사전 검증과 명확한 확인 UX를 둔다.
7. 기존 Phase 1~3 기능의 도메인 경계를 깨지 않는다.

## Phase 4A: 운영 대시보드

### 목표
관리자가 앱 첫 화면에서 전체 운영 상태를 빠르게 파악할 수 있게 합니다.

운영 대시보드는 새로운 업무 기능이 아니라, 기존 기능들의 상태를 모아 보여주는 관제 화면입니다.

### 주요 사용자 질문

- 오늘 처리된 공고, 문서, 판단 실행 수는 몇 개인가?
- 실패한 작업이 있는가?
- OCR/API/DB/storage 상태는 정상인가?
- 검토 대기 중인 항목은 몇 개인가?
- 백업이 오래되지 않았는가?

### UX 구성

권장 라우트:

- `/operations`

화면 구성:

- 상단 상태 요약
  - 전체 상태: 정상 / 주의 / 조치 필요
  - 마지막 데이터 백업 시각
  - 최근 실패 작업 수
  - 검토 대기 항목 수
- 작업 상태 카드
  - 일반 문서 분석
  - 법인 증빙자료 분석
  - 기준문서 처리/인덱싱
  - 기준 규칙 후보 검토
  - 나라장터 수집
  - 판단 실행
- 최근 실패 목록
  - 작업 유형
  - 대상 이름
  - 실패 시각
  - 짧은 실패 사유
  - 상세 보기 버튼
- 검토 대기 목록
  - 기준 규칙 후보 `needs_review`
  - 판단 실행 `review_status=pending`
  - 증빙자료 후보 승인 대기
- 빠른 이동 버튼
  - 나라장터 수집 이력
  - 판단 실행 이력
  - 기준 규칙 후보 관리
  - 백업/복원

### 데이터 조회 기준

우선은 별도 테이블을 만들지 않고 기존 테이블을 집계합니다.

- `corporation_evidence_documents`
- `basis_documents`
- `basis_rule_candidates`
- `nara_notices`
- `nara_collection_runs`
- `judgment_runs`
- `integration_test_results`

추후 Phase 4B에서 공통 작업 테이블이 생기면 대시보드는 그 테이블도 함께 집계합니다.

### 테스트 계획

- API 테스트
  - 빈 DB에서도 `GET /api/operations/summary`가 200을 반환한다.
  - 실패 작업이 있으면 `overall_status=warning` 또는 `action_required`가 된다.
  - API 키 원문은 응답에 노출되지 않는다.
  - OCR 미설치 상태도 장애가 아니라 `unavailable` 상태로 표시된다.
- 프론트엔드 테스트 또는 빌드 검증
  - 운영 대시보드 라우트가 빌드된다.
  - 빈 상태, 정상 상태, 실패 상태가 모두 깨지지 않는다.

### 완료 기준

- 관리자가 `/operations`에서 최근 실패와 검토 대기 항목을 확인할 수 있다.
- 민감한 설정값은 마스킹된다.
- 대시보드 조회가 기존 처리 파이프라인을 실행하지 않는다.
- 전체 백엔드 테스트와 프론트엔드 빌드가 통과한다.

## Phase 4B + 4C: 작업/실패 관리

### 목표
PDF 분석, OCR, 기준문서 처리, 나라장터 수집, 판단 실행처럼 시간이 걸리거나 실패할 수 있는 작업을 운영자가 추적하고 재시도할 수 있게 합니다.

Phase 4B는 작업 실행/이력 관리이고, Phase 4C는 실패 사유/재시도 관리입니다. 실제 UX와 데이터 모델은 함께 설계하는 것이 자연스럽습니다.

### 현재 문제

현재 일부 작업은 개별 테이블에 상태가 남습니다.

- `nara_collection_runs`
- `judgment_runs`
- `basis_documents.processing_status`
- `corporation_evidence_documents.analysis_status`
- `nara_notices.analysis_status`

하지만 운영자 관점에서는 작업 유형이 달라도 다음 정보가 공통으로 필요합니다.

- 언제 시작했는가?
- 언제 끝났는가?
- 성공/실패/부분 실패인가?
- 실패 사유는 무엇인가?
- 다시 실행할 수 있는가?
- 결과가 어디에 저장되었는가?

### 제안 데이터 모델

신규 테이블 후보:

```sql
CREATE TABLE operation_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  operation_type TEXT NOT NULL,
  target_type TEXT DEFAULT '',
  target_id INTEGER,
  status TEXT NOT NULL,
  requested_by TEXT DEFAULT 'local_admin',
  request_json TEXT DEFAULT '{}',
  result_json TEXT DEFAULT '{}',
  error_message TEXT DEFAULT '',
  error_code TEXT DEFAULT '',
  retry_of_run_id INTEGER,
  retry_count INTEGER DEFAULT 0,
  started_at TEXT,
  finished_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

상태값:

- `queued`
- `running`
- `completed`
- `partial_failed`
- `failed`
- `cancelled`

작업 유형:

- `document_analysis`
- `corporation_evidence_analysis`
- `basis_document_processing`
- `basis_rule_candidate_extraction`
- `nara_notice_analysis`
- `nara_collection`
- `judgment_run`
- `backup_create`
- `backup_restore`

### 기존 테이블과의 관계

초기 구현에서는 모든 기존 작업을 한 번에 이전하지 않습니다.

권장 순서:

1. `nara_collection_runs`와 `judgment_runs`를 `operation_runs`에 미러링한다.
2. 기준문서 재처리와 기준 규칙 후보 추출을 `operation_runs`에 기록한다.
3. 증빙자료 분석과 일반 문서 분석은 이후 확장한다.

이 방식이면 기존 API/UX를 크게 흔들지 않고 운영 이력 통합을 시작할 수 있습니다.

### UX 구성

권장 라우트:

- `/operation-runs`
- `/operation-runs/:id`

목록 화면:

- 필터
  - 작업 유형
  - 상태
  - 기간
  - 실패만 보기
  - 대상 유형
- 테이블 컬럼
  - 상태
  - 작업 유형
  - 대상
  - 시작 시각
  - 종료 시각
  - 소요 시간
  - 실패 사유 요약
  - 재시도 가능 여부

상세 화면:

- 작업 기본 정보
- 요청 payload 요약
- 결과 payload 요약
- 실패 사유
- 관련 대상 바로가기
- 재시도 버튼
- 동일 대상의 과거 실행 이력

### 재시도 정책

초기 정책:

- 자동 재시도는 구현하지 않는다.
- 관리자가 실패 사유를 보고 수동 재시도한다.
- 재시도는 원본 `operation_run`을 덮어쓰지 않고 새 run으로 만든다.
- 새 run은 `retry_of_run_id`로 원본 run과 연결한다.
- 재시도 시 기존 결과를 삭제하는 작업은 명확한 확인 문구를 둔다.

재시도 가능 예시:

- 나라장터 수집 실패
- 공고 첨부 다운로드 실패
- 기준문서 재처리 실패
- 기준 규칙 후보 추출 실패
- 판단 실행 실패

재시도 제한 예시:

- 원본 파일이 삭제된 문서
- API 키 미설정 상태
- 백업 파일이 없는 복원 작업

### 백엔드 API 제안

신규 API:

- `GET /api/operation-runs`
- `GET /api/operation-runs/:id`
- `POST /api/operation-runs/:id/retry`

보조 내부 서비스:

- `create_operation_run(...)`
- `mark_operation_run_running(...)`
- `mark_operation_run_completed(...)`
- `mark_operation_run_failed(...)`
- `retry_operation_run(...)`

### 실패 사유 표준화

`error_code` 후보:

- `missing_api_key`
- `network_error`
- `download_failed`
- `unsupported_file_type`
- `parser_failed`
- `ocr_unavailable`
- `ocr_failed`
- `ai_provider_failed`
- `database_error`
- `storage_error`
- `validation_error`
- `unknown_error`

관리자 화면에는 `error_code`와 `error_message`를 함께 표시하되, 민감한 원문 payload나 API 키는 숨깁니다.

### 테스트 계획

- API 테스트
  - 작업 run 생성/상태 전환이 기록된다.
  - 실패 run은 `error_code`, `error_message`를 가진다.
  - 재시도는 기존 run을 수정하지 않고 새 run을 만든다.
  - API 키 원문이나 민감 식별정보가 `result_json`/`error_message`에 노출되지 않는다.
- 통합 테스트
  - 나라장터 수집 실패가 `operation_runs`에 남는다.
  - 판단 실행 성공이 `operation_runs`에 남는다.
  - 기준문서 재처리 실패가 `operation_runs`에 남는다.
- UX 검증
  - 실패만 보기 필터가 동작한다.
  - 상세 화면에서 관련 대상으로 이동할 수 있다.
  - 재시도 버튼은 재시도 불가능 상태에서 비활성화된다.

### 완료 기준

- 관리자가 작업 이력 목록에서 성공/실패/부분 실패를 구분할 수 있다.
- 실패 상세에서 원인과 다음 행동을 볼 수 있다.
- 재시도는 새 실행 이력으로 남는다.
- 기존 `nara_collection_runs`, `judgment_runs` 기능이 깨지지 않는다.

## Phase 4D: 백업/복원

### 목표
로컬 PC 운영 중 데이터 손실을 줄이기 위해 DB, 업로드 파일, 기준문서 인덱스, 설정 샘플을 하나의 백업 단위로 묶고 복원할 수 있게 합니다.

### 백업 대상

필수:

- SQLite DB 파일
- `storage/uploads/`
- `storage/corporation-evidence/`
- `storage/basis/`
- `storage/basis-index/`
- `storage/nara-notices/`

선택:

- PDF 테스트 샘플 캐시
- QA manifest
- 로컬 로그 파일

제외:

- API 키 원문이 들어 있는 `.env`
- `node_modules`
- 프론트엔드 `dist`
- 임시 작업 폴더

### 백업 파일 구조

권장 파일명:

```text
smart-procurement-backup-YYYYMMDD-HHMMSS.zip
```

ZIP 내부 구조:

```text
manifest.json
database/app.db
storage/uploads/...
```

### 복원 정책

복원은 파괴적인 작업이므로 다음 안전장치를 둡니다.

- 복원 전 현재 DB와 storage를 자동으로 safety backup 한다.
- 복원 파일의 `manifest.json`을 먼저 검증한다.
- app 실행 중 복원으로 파일 충돌이 발생할 수 있으므로, 초기 구현은 “복원 준비 패키지 생성”까지 허용하고 실제 교체는 재시작 안내로 처리할 수 있다.
- 직접 교체 복원은 Phase 4D 후반 작업으로 둔다.

권장 구현 순서:

1. 백업 생성
2. 백업 검증
3. 복원 dry-run
4. 현재 데이터 safety backup
5. 복원 실행
6. 복원 후 무결성 확인

# AI / Engineering Version (English)

## Purpose
This document defines the Phase 4 operational productization scope for the `SMART Procurement Calculator`.

Phase 4 is not primarily about improving the judgment engine. It is about making the local single-PC product operable: status visibility, run tracking, failure recovery, backup, and restore readiness.

This plan covers:

- Phase 4A: Operations dashboard
- Phase 4B + 4C: Operation run and failure management
- Phase 4D: Backup and restore

## Baseline
As of 2026-05-24, the product already has:

- project document upload and analysis
- corporation evidence upload, extraction, classification, review, and profile enrichment
- Nara API-based notice search, save, attachment download, and summarization
- notice requirement extraction and corporation gap preview
- basis document upload, parsing, OCR degradation, chunking, local indexing, and search
- basis rule candidate extraction, approval, rejection, and editing
- Phase 3 gap-first judgment runs, citation candidates, preparation guides, and review state
- Nara collection run history
- cached PDF sample QA flow

Phase 4 should answer operational questions:

- Is the system healthy?
- What failed recently?
- Why did it fail and can it be retried?
- Is critical data backed up?
- Can a backup be validated before restore?

## Principles

1. Preserve local single-PC operation.
2. Keep auth, multi-user, and cloud deployment out of the default Phase 4 scope.
3. Surface failures clearly.
4. Prefer manual retry with visible context before automatic retry.
5. Back up both SQLite and storage files.
6. Treat restore as destructive; validate first and require clear confirmation.
7. Do not break Phase 1-3 domain boundaries.

## Phase 4A: Operations Dashboard

### Goal
Give the administrator one screen to understand current operational status.

Suggested route:

- `/operations`

Suggested API:

- `GET /api/operations/summary`

The dashboard aggregates existing tables first:

- `corporation_evidence_documents`
- `basis_documents`
- `basis_rule_candidates`
- `nara_notices`
- `nara_collection_runs`
- `judgment_runs`
- `integration_test_results`

Expected widgets:

- overall health
- last backup status
- recent failures
- review queues
- operation counts for the last 24 hours

Acceptance criteria:

- The dashboard works with an empty database.

- The dashboard does not trigger processing pipelines.

## Phase 4B + 4C: Operation Run And Failure Management

### Goal
Track long-running or failure-prone tasks consistently and let the administrator inspect and retry failures.

Candidate table:

```sql
CREATE TABLE operation_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  operation_type TEXT NOT NULL,
  target_type TEXT DEFAULT '',
  target_id INTEGER,
  status TEXT NOT NULL,
  requested_by TEXT DEFAULT 'local_admin',
  request_json TEXT DEFAULT '{}',
  result_json TEXT DEFAULT '{}',
  error_message TEXT DEFAULT '',
  error_code TEXT DEFAULT '',
  retry_of_run_id INTEGER,
  retry_count INTEGER DEFAULT 0,
  started_at TEXT,
  finished_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

Statuses:

- `queued`
- `running`
- `completed`
- `partial_failed`
- `failed`
- `cancelled`

Initial operation types:

- `nara_collection`
- `judgment_run`
- `basis_document_processing`
- `basis_rule_candidate_extraction`
- `nara_notice_analysis`

Suggested APIs:

- `GET /api/operation-runs`
- `GET /api/operation-runs/:id`
- `POST /api/operation-runs/:id/retry`

Implementation order:

1. Add `operation_runs` table and service helpers.
2. Mirror `nara_collection_runs` and `judgment_runs` into `operation_runs`.
3. Record basis document reprocessing and basis rule extraction.
4. Add failure detail UX and manual retry UX.

Retry policy:

- Do not overwrite the original failed run.
- Create a new run and link it with `retry_of_run_id`.
- Disable retry when the target is missing or required configuration is absent.
- Never expose raw secrets in `request_json`, `result_json`, or `error_message`.

Acceptance criteria:

- The administrator can filter run history by status, type, and time range.
- Failure details include standardized `error_code` and safe `error_message`.
- Retry creates a new run.
- Existing `nara_collection_runs` and `judgment_runs` behavior remains compatible.

## Phase 4D: Backup And Restore

### Goal
Create, validate, and eventually restore full local backups containing SQLite data and storage assets.

Required backup contents:

- SQLite database file
- `storage/uploads/`
- `storage/corporation-evidence/`
- `storage/basis/`
- `storage/basis-index/`
- `storage/nara-notices/`

Excluded:

- `.env`
- raw API keys
- `node_modules`
- frontend `dist`
- temp directories

Backup filename:

```text
smart-procurement-backup-YYYYMMDD-HHMMSS.zip
```

Candidate table:

```sql
CREATE TABLE backup_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  backup_type TEXT NOT NULL,
  status TEXT NOT NULL,
  file_name TEXT DEFAULT '',
  file_path TEXT DEFAULT '',
  file_size_bytes INTEGER DEFAULT 0,
  manifest_json TEXT DEFAULT '{}',
  validation_json TEXT DEFAULT '{}',
  error_message TEXT DEFAULT '',
  created_at TEXT NOT NULL,
  completed_at TEXT,
  updated_at TEXT NOT NULL
);
```

Suggested APIs:

- `GET /api/backups`
- `POST /api/backups`
- `GET /api/backups/:id`
- `POST /api/backups/validate`
- `POST /api/backups/restore-plan`
- `POST /api/backups/:id/restore`

Restore should begin with dry-run validation:

1. Validate `manifest.json`.
2. Validate required files.
3. Validate checksums.
4. Create a safety backup of current data.
5. Execute restore only after explicit confirmation.

Acceptance criteria:

- The administrator can create a backup from the UI.
- The backup ZIP contains DB and required storage paths.
- `.env` and raw API keys are excluded.
- Backups can be validated.
- Restore can at least run as dry-run safely.
