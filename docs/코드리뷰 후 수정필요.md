## 이번 수정계획 대상

### 1. P1 기준문서 재처리 swap 원자성 보강

수정 상태:
- [x] 2026-05-31 코드 수정 및 회귀 테스트 추가 완료

문제:
- 기준문서 재처리 중 새 청크/인덱스를 만든 뒤 기존 청크/기존 인덱스를 삭제하는 흐름이 있다.
- 최종 문서 상태 업데이트 전에 예외가 발생하면 기존 정상 기준 지식이 일부 삭제되거나, DB 청크와 JSON 인덱스가 서로 어긋날 수 있다.
- 실패한 재처리 결과가 기존 정상 지식을 손상시키지 않아야 한다.

수정 방향:
- 기존 청크/인덱스를 바로 삭제하지 않고, 새 처리 결과를 임시 run으로 끝까지 생성한다.
- 새 run의 청크 DB insert, 임시 인덱스 payload 생성, 문서 상태 업데이트에 필요한 metadata 계산을 먼저 완료한다.
- 모든 검증이 끝난 뒤 마지막 단계에서만 기존 run을 비활성/삭제하고 새 run을 최신 결과로 확정한다.
- 파일 기반 JSON 인덱스는 임시 파일에 먼저 기록한 뒤 `replace`로 교체한다.
- 예외 발생 시 새 run 청크와 새 run 인덱스만 정리하고 기존 run 청크/인덱스/승인 후보는 유지한다.

세부 작업:
1. `basis_document.py`에 최신 run 확정 전용 helper를 분리한다.
2. 인덱스 저장 흐름을 `build new payload -> write temp -> atomic replace` 구조로 정리한다.
3. 기존 청크 삭제와 기존 벡터 삭제는 최종 확정 구간에서만 수행한다.
4. 최종 확정 전 예외가 나도 기존 청크/기존 벡터가 유지되는지 보장한다.
5. 승인된 `basis_rule_candidates`는 재처리 성공 시에만 `needs_review`로 되돌린다.

추가 테스트:
- 새 run 인덱싱 성공 후 기존 청크 삭제 직전/직후에 예외가 발생해도 기존 청크와 기존 후보가 유지되는 테스트
- JSON 인덱스 저장 실패 시 기존 인덱스가 유지되는 테스트
- 재처리 성공 시 기존 청크가 새 청크로 교체되고 승인 후보가 재검토 상태가 되는 테스트

완료 기준:
- 재처리 실패 시 기존 검색 결과와 기존 승인 후보가 유지된다.
- 재처리 성공 시에만 청크/인덱스/citation 후보 상태가 새 결과 기준으로 바뀐다.
- 전체 백엔드 테스트가 통과한다.

### 2. P2 나라장터 재분석 첨부 없음/지원 첨부 없음 처리 보강

수정 상태:
- [x] 2026-05-31 코드 수정 및 회귀 테스트 추가 완료

문제:
- 기존 공고 재분석에서 새 요청의 첨부가 없거나 지원 가능한 첨부가 0개여도 `failed_count == 0`이면 분석 성공으로 간주될 수 있다.
- 이 경우 기존 PDF 기반 첨부, 요구조건, 비교 결과, 판단 이력이 삭제될 수 있다.
- 나라장터 상세 API가 일시적으로 첨부를 누락하거나, 지원 제외 파일만 내려주는 경우 기존 정상 분석 결과를 보존해야 한다.

수정 방향:
- 기존 결과가 있는 공고의 재분석에서는 `completed` 교체 조건을 더 엄격하게 둔다.
- 지원 가능한 첨부가 0개인 재분석은 기존 결과가 있으면 결과 교체 대상이 아니라 `partial_failed` 또는 `no_supported_attachments_preserved` 성격으로 처리한다.
- 신규 공고는 기존처럼 첨부가 없어도 공고 메타데이터 기반 요약/요구조건 추출을 허용할 수 있으나, 기존 공고 재분석에서는 기존 결과를 지우지 않는다.

세부 작업:
1. `previous_has_results`가 true인 경우 `supported_count == 0`이면 `replace_existing_results=False`로 처리한다.
2. 기존 결과 보존 시 `analysis_summary_json`, `analysis_summary_markdown`, 기존 첨부, 요구조건, 비교, 판단 이력을 유지한다.
3. 상태와 오류 메시지는 “재분석에서 지원 가능한 첨부가 없어 기존 분석 결과를 유지했습니다.”처럼 관리자에게 명확히 남긴다.
4. 신규 공고와 기존 공고의 상태 정책을 분리해 테스트한다.

추가 테스트:
- 기존 첨부/요구조건이 있는 공고를 첨부 없는 payload로 재분석해도 기존 결과가 유지되는 테스트
- 기존 첨부/요구조건이 있는 공고를 HWP 등 지원 제외 첨부만 있는 payload로 재분석해도 기존 결과가 유지되는 테스트
- 신규 공고는 첨부가 없어도 현재 정책대로 저장/요약 가능한지 확인하는 회귀 테스트

완료 기준:
- 기존 결과가 있는 공고는 새 재분석이 충분한 첨부/분석 결과를 만들었을 때만 교체된다.
- 첨부 누락/지원 제외 첨부만 있는 재분석은 기존 결과를 보존한다.
- 전체 백엔드 테스트가 통과한다.

## 이번 작업 범위 제외, 수정 필요 기록만

### A. P1 첨부 URL 검증 DNS rebinding/shared address 보강

기록:
- 현재 첨부 URL 검증은 사전 DNS 해석으로 내부망 IP를 차단하지만, 실제 다운로드 시점에 DNS가 다시 해석될 수 있다.
- `100.64.0.0/10` 같은 shared address 대역도 공개 인터넷 주소로 취급하면 안 된다.

후속 수정 필요:
- `ipaddress.is_global` 중심의 allow 정책으로 전환 검토
- 검증된 IP로 직접 연결하거나, 커스텀 resolver/connection 정책으로 DNS rebinding 완화 검토
- redirect마다 동일 정책 적용 유지
- 관련 테스트 추가

### B. P2 백업 dry-run restore API 경로 제한 일관화

수정 상태:
- [x] 2026-05-31 전체 코드리뷰/테스트 보강 중 수정 및 회귀 테스트 추가 완료

기록:
- `/api/backups/validate`, `/api/backups/restore-plan`은 백업 경로 제한 helper를 사용한다.

후속 수정 필요:
- dry-run restore API도 `_resolve_allowed_backup_file_path()` 또는 backup id 기반 검증을 사용하도록 통일
- DB 값 오염 시 `storage/backups` 밖 파일 검증 결과가 노출되지 않도록 보강
- 관련 테스트 추가

# AI / Engineering Version (English)

## Purpose

Track post-review remediation items. Only two items are selected for immediate fix planning. Remaining findings are recorded as backlog items and should not be implemented in the immediate task unless explicitly requested.

## Immediate Fix Plan

### 1. P1 Atomic Basis Document Reprocessing Swap

Status:
- [x] 2026-05-31 implementation and regression tests completed

Issue:
- Reprocessing can create new chunks/index entries and then delete old chunks/index entries before the final document update is fully committed.
- If an exception happens between deletion and final completion, old basis knowledge may be partially destroyed or DB chunks may diverge from the JSON index.

Plan:
- Treat each reprocess as a temporary run until all new artifacts are complete.
- Build new chunks, new index payload, and final metadata before deleting old artifacts.
- Only after all validation passes, atomically replace the JSON index file and switch DB rows to the new run.
- On any exception, remove only the new temporary run artifacts and preserve old chunks, old index entries, and reviewed rule candidates.
- Mark approved rule candidates as `needs_review` only after a successful reprocess.

Tests:
- Failure after new indexing but before final swap preserves old chunks and candidates.
- JSON index save failure preserves the old index.
- Successful reprocess replaces old chunks and marks candidates for revalidation.

Acceptance Criteria:
- Failed reprocess leaves prior search/citation knowledge intact.
- Successful reprocess is the only path that changes active chunks/index/candidate review state.

### 2. P2 Preserve Existing Nara Analysis When Reanalysis Has No Supported Attachments

Status:
- [x] 2026-05-31 implementation and regression tests completed

Issue:
- Reanalysis with no attachments or no supported attachments can be treated as completed because `failed_count == 0`.
- This can delete existing PDF-backed attachments, requirements, comparisons, and judgment runs.

Plan:
- If an existing notice already has analysis results and the new run has `supported_count == 0`, do not replace existing results.
- Preserve existing attachments, requirements, comparisons, judgments, summary JSON, and summary markdown.
- Record a clear status/error message explaining that existing analysis was preserved because the reanalysis had no supported attachment.
- Keep new notice behavior separate from existing notice reanalysis behavior.

Tests:
- Existing notice reanalysis with no attachments preserves prior results.
- Existing notice reanalysis with unsupported attachments only preserves prior results.
- New notice without attachments still follows the current metadata-based save/summary policy.

Acceptance Criteria:
- Existing notice results are replaced only when the new reanalysis has sufficient supported input and completes.
- Missing/unsupported attachments do not erase existing analysis.
